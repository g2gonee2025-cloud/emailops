# Code Style (EmailOps)

We follow **PEP 8** for Python naming and formatting with explicit rules:

## Naming
- **Modules & files:** `snake_case.py`
- **Classes:** `PascalCase`
- **Functions & Methods:** `snake_case`
- **Variables & Attributes:** `snake_case`
- **Constants:** `UPPER_SNAKE_CASE`
- **Private/Internal:** prefix with `_` (e.g., `_cache_size`)

### Qt/PyQt Exceptions
Qt's event system and properties require specific **camelCase** names to override framework methods or expose `Q_PROPERTY` values. We intentionally keep these names and document them inline:

- Event handlers: `enterEvent`, `leaveEvent`, `mousePressEvent`, `paintEvent`, `closeEvent`, etc.
- Animated property used by `QPropertyAnimation`: `animationProgress`

In each of these cases, the source includes a comment such as:

```py
# Qt override: method name must remain camelCase for Qt event system
def paintEvent(self, event):
    ...
```

## Parameters
- Use descriptive names. We standardized autogenerated placeholders like `a0` to `event`.

## Linting
We provide a Ruff config (see `pyproject.toml`) that ignores naming warnings for the Qt-specific methods listed above.
